Identification & AAA

Identification, Authentication, Authorization and Accounting (AAA).

  • A methodology in which users identify themselves to a system and are required to authenticate their identity with some form of password, token or biometric.
  • Once authenticated, a system must monitor user actions, only allowing actions which a user is authorized to make and recording user actions in a process known as accounting.
  • Used to create frameworks for access control in which a server verifies the authentication and authorization of entities that require network access and manages their accounts.

Multifactor Authentication

Authentication using two or more factors to achieve authentication.

Using multiple forms of authentication from only one factor is not multifactor authentication.

    Authentication Factors:

  • Something you know (password/pin)
  • Something you have (key,token,smartcard)
  • Something you are (iris scan, fingerprinting)
  • Something you do (gait analysis)
  • Somewhere you are (location tracking)

Identity and Access Management

There are different methods for managing an identification and authentication scheme, and organizations can even work together to allow mutual access to resources to authenticated users.
Using the following IAM concepts can make life easier for end user’s while maintaining security within an organization.

Federation

A collection of domains (sometimes separate companies) that have established trust among themselves. The level of trust may vary, but typically includes authentication and may include authorization.

For example, logging into Twitter with your Google account.

Single Sign-On and Single Sign-Off

Single Sign-On (SSO) enables a user to authenticate to a central authorization server to gain access to all the applications or machines which they are permitted to use.
Single Sign-Off enables a user to log out of multiple applications/systems with a single action.

Transitive Trust

Allows the system to trust networks which are trusted by the original domain without explicitly creating this trust.
If Domain 1 trusts Domain 2, and Domain 2 trusts Domain 3, Domain1 would also trust Domain 3, based on Domain 2’s trust.
